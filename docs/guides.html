Integration Guide

Implementation guidance for Hybrid Client API Part B in production environments.

Error Handling #

Always return JSON responses with predictable fields for failures:

JSON
{
  "success": false,
  "code": "HC_404_001",
  "message": "User not found"
}
HTTP Code Meaning
402 HC_402_001 Insufficient points
404 HC_404_001 User not found
500 HC_500_001 Internal server error
503 HC_503_001 Service unavailable

Timeouts and Retry #

Reward Geni timeout and retry behavior from the Part B spec:

Endpoint Timeout Retry
GET /v1/points/:externalUserId 10 seconds Up to 2 retries on network errors
POST /v1/redeem 15 seconds Up to 2 retries on network errors

Recommended Retry Logic on Your API Side

Prefer fast failures over long blocking waits. Keep responses within timeout windows and return 503 when downstream systems are unavailable.

JavaScript
function serviceUnavailable(message) {
  return {
    status: 503,
    body: {
      success: false,
      code: 'HC_503_001',
      message
    }
  };
}

// Example usage:
// return serviceUnavailable('Redemption service temporarily unavailable. Please try again later.');

Idempotency and Consistency #

Use orderId from the redeem request as an idempotency key to avoid duplicate deductions for retried requests.

  • If the same orderId arrives again, return the original successful result.
  • Store and return your transactionId consistently.
  • Ensure remainingPoints reflects the final ledger value after deduction.
